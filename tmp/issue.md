## 概要

本Epicは、PoohMaの「家族間で安全かつ手軽にアカウント情報を共有・管理する」というコアバリューを最大化し、本番環境での安全な運用を担保するための包括的ロードマップです。
セキュリティ監査による脆弱性の修正、クライアントサイドE2EEのアーキテクチャ堅牢化、モバイルUI/UXの最適化、そして日常的な利便性を高める機能拡張を、4つのフェーズに分けて段階的に追跡・実装します。

## 🛡️ Phase 1: セキュリティ・アーキテクチャの堅牢化 (Security & Architecture)

E2EEの前提を揺るがす致命的な脆弱性の修正と、データ整合性を担保するためのバックエンド改修、および品質の劣化を防ぐ自動チェック体制を整備します。

- [x] **`sessionStorage` へのマスターキーキャッシュ・永続化の即刻廃止【最優先】**
  - **課題:** `sessionStorage` に生のマスターキー（Base64等）を保存しているため、万が一XSSやサプライチェーン攻撃が発生した場合、E2EEが破られ全データが窃取される。
  - **対策:** 永続化を廃止し、キーはReactのState（メモリ内）のみで保持するか、Web Worker内に隔離してメインスレッドに露出させない設計へ刷新する（`extractable: false` の徹底）。
- [x] **OGP取得処理における DNS Rebinding (TOCTOU) SSRF 脆弱性の修正【最優先】**
  - **課題:** URLのDNS検証直後に元のURLで `fetch` を行うため、攻撃者のDNS TTL操作により内部ネットワーク（`127.0.0.1` 等）へのSSRF攻撃が可能。
  - **対策:** 検証を通過した「IPアドレスそのもの」に対して直接リクエストを送信（Hostヘッダーにドメインを指定）する。また、ReDoSリスクのある正規表現HTMLパーサーを `cheerio` 等に移行する。
- [x] **OGP取得処理（SSRF対策）におけるリダイレクト・リソース制限の強化**
  - **課題:** 現在実装されている安全なバッファ取得ロジックにおいて、悪意あるリダイレクトの無限ループや、超巨大レスポンスによるサーバーリソースの枯渇攻撃（Slowloris/DoS）への明示的な防御線がない。
  - **対策:** `fetchSafeBuffer` 内のリダイレクト追跡回数に上限を設定し、レスポンスサイズが最大値（例: 5MB）を超えた場合は即座にストリームを遮断する制限を追加する。
- [x] **Convex関数における手動認可チェックの自動化・共通化 (IDOR対策)**
  - **課題:** 各関数で個別に `ctx.auth.getUserIdentity()` 等を行っており、書き忘れによる他者データの不正改ざん（IDOR）リスクがある。
  - **対策:** ConvexのCustom Functionsを用いて `authenticatedMutation` や `familyBoundQuery` のようなラッパーを作成し、認証・所属家族のスコープバインドを強制・自動化する。
- [ ] **家族変更・鍵ローテーション時の一貫性破綻リスク解消**
  - **課題:** 鍵変更時にクライアントサイドで全レコードを再暗号化しバルク送信しているため、通信切断やメモリ不足時に一部データだけが古い鍵のまま取り残されデータロストする。
  - **対策:** 各レコードに鍵バージョンを持たせ、閲覧・編集時に個別に再暗号化するオンデマンド方式、またはレジューム可能なトランザクション設計へ変更する。
- [ ] **Convex「配列埋め込み（Embedded Array）」の分離**
  - **課題:** `credentials` を配列で埋め込み、都度UUIDを再生成して丸ごと置換しているため、同時編集時の楽観的ロック（OCC）競合によるデータロストやUIのフォーカス崩れが発生する。
  - **対策:** `credentials` を独立したテーブルに切り出し、 `recordId` による1:Nの参照関係を構築する。
- [x] **エンベロープ暗号（鍵階層化）の導入による鍵管理の効率化**
  - **課題:** マスターキー（KEK）でレコードを直接暗号化しているため、マスターキーの変更やローテーションを行う際に全データレコードの再暗号化・再書き込みが発生し、スケール時の負荷とリスクが極めて高い。
  - **対策:** 各レコード専用のランダムな「データ暗号化鍵（DEK）」を生成してデータを暗号化し、そのDEKをマスターキーでラップしてレコードと共に保存する階層型鍵管理（エンベロープ暗号）へ移行する。
- [ ] **PBKDF2のイテレーション回数の動的（スキーマ）管理化**
  - **課題:** 現在 `crypto.ts` 内で定数 `300000` がハードコードされている。将来的なセキュリティ強度引き上げのためにこの値を変更すると、既存ユーザーが古いマスターキーを二度と復号できなくなる（後方互換性の喪失）。
  - **対策:** `families` テーブルに `cryptoVersion` や `kdfIterations` フィールドを追加し、鍵作成時の回数を記録。復号時はDBから取得した値を動的に適用する設計に変更し、将来のアルゴリズム強化に備える。
- [x] **E2EE暗号化データのスキーマバリデーション強化**
  - **課題:** 現在のZod相関チェック（`superRefine`）は優れているが、Base64の正規表現検証のみであり、暗号データの整合性やIVの正確な長さ（12バイト/Base64で16文字）が事前に厳密に担保されていない。
  - **対策:** `iv` + `ciphertext` + `tag` を一つの認証付き暗号（AEAD）データスキーマとして定義し、バックエンドへ送信される前にクライアント側で厳格に長さと整合性を検証する。
- [x] **CSV入出力時のサニタイズ (CSV Injection) とメインスレッド・ブロッキング (DoS) 対策**
  - **課題:** 悪意ある数式（`=cmd|...`）がエクスポート時に実行されるリスク。また、大量レコードのインポート/暗号化でメインスレッドがフリーズ（Jank）する。
  - **対策:** エクスポート時に数式トリガー文字をシングルクォートでエスケープする。インポート時はZodの `.max()` 制限を厳格化し、処理をチャンク分割または Web Worker にオフロードする。
- [ ] **厳格な CSP (Content Security Policy) の導入**
  - **課題:** XSS発生時のデータ外部送信（Exfiltration）を防ぐ防衛線がない。
  - **対策:** `script-src 'self' 'nonce-...'` や `connect-src` を設定し、許可されたバックエンド以外への通信をブラウザレベルで強制遮断する。
- [x] **状態管理とキャッシュの責務分離（TanStack Query vs Convex）**
  - **課題:** 両者のキャッシュ機構が混在し、無駄な二重フェッチや不整合の原因になっている。
  - **対策:** 責務の境界線を明確に設計・ドキュメント化し、将来的にはConvexへの統合を検討する。
- [x] **移行ドキュメントのクリーンアップと Prisma の完全削除**
  - **課題:** 旧Prisma環境とConvex環境のドキュメント・依存関係が混在している。
  - **対策:** `code-design.md` をConvex準拠に更新し、 `@prisma/client` 等の未使用パッケージ・コードを物理削除する。
- [x] **Zodスキーマによる大容量データ入力制限の厳格化**
  - **課題:** `passwordHint` や `memo` に文字数上限が定義されていないため、巨大なテキストが入力された場合にConvexのドキュメントサイズ制限への抵触や、クライアント側でのAES-GCM暗号化・復号時のメモリ枯渇によるUIフリーズ（DoS状態）を招くリスクがある。
  - **対策:** `src/utils/schemas.ts` にて `memo`（最大10,000文字）、`passwordHint`（最大2,000文字）などの厳格な上限を定義し、アプリの安定性をフロント・バック両レイヤーで担保する。
- [x] **本番ビルドプロセスおよびテストカバレッジ閾値のCI組み込み**
  - **課題:** CIで `build` やカバレッジが検証されていないため、TanStack Routerの自動生成型の不整合やSSR固有のエラーが本番デプロイ時に初めて発覚するリスクがある。また、暗号コアロジックの劣化（デグレード）を自動検知できない。
  - **対策:** CIステップに `pnpm run build` を追加。さらに `vitest.config.ts` にてコアロジック（`src/lib/crypto.ts` や RLS周辺）のカバレッジ最低閾値（例: 全体80%以上）を設定し、下回るプルリクエストを自動ブロックする。

## 📱 Phase 2: モバイルUI/UX・アクセシビリティの抜本的改善 (Mobile UX & A11y)

スマートフォンでの片手操作や、ITに詳しくない家族でも直感的に操作でき、「迷わせない・ストレスを与えない」デザインへの最適化を行います。

- [ ] **`PasscodeProvider` における同時呼び出し競合（Race Condition）の修正**
  - **課題:** `requireUnlock` が複数のコンポーネントからほぼ同時に呼び出された場合、`resolvePromiseRef.current` が後続の処理で上書きされてしまい、先行するプロンプトのPromiseが解決されずUIがハング（ローディングが回り続ける等）する。
  - **対策:** すでにアンロックプロンプトが開いている場合はPromiseの上書きをブロックし、現在待機中の同一Promiseを返却（シングルトン化）するか、リゾルバをキュー（Queue）で管理する仕組みへリファクタリングする。
- [ ] **ネスト（階層型）ドロップダウンメニューの廃止**
  - **課題:** スマホ画面で入れ子メニュー（「データ管理 ＞」等）の小さな矢印タップは誤操作を誘発する。
  - **対策:** モバイル環境では「ボトムシート（Bottom Sheet）」またはフルスクリーンの設定画面へ置き換える。
- [ ] **「+ 新規登録」ボタン（主要アクション）の再配置**
  - **課題:** 画面右上の配置は、縦長スマホの片手操作時に親指が最も届きにくい。
  - **対策:** 画面右下に固定配置される Floating Action Button (FAB) を導入する。
- [ ] **フォーム入力欄のコントラスト不足とアフォーダンス改善**
  - **課題:** Geistデザインの過剰適用により入力欄の境界がなく、タップ可能領域が不明瞭。Danger Zoneの赤文字も視認性が低い。
  - **対策:** 入力欄に明確な境界線や背景色を設け、Danger Zoneは明確な赤背景のアウトラインボタン等に変更してWCAGコントラスト比を担保する。
- [ ] **タッチターゲット（タップ判定領域）の極小化解消**
  - **課題:** コピーボタンや「← 戻る」がテキスト直書きであり、タップ領域がApple/Googleのガイドライン（44x44px）を満たしていない。
  - **対策:** 独立したアイコン付きボタンコンポーネント化し、適切なパディングを確保する。
- [ ] **マイクロコピー（用語・カラー）の統一とエラーUIの改善**
  - **課題:** 「自分のみ」と「個人」の表記揺れ。エラー時にモノクロトーストが表示され冷徹な印象を与える。
  - **対策:** 用語を統一し、セマンティックカラー（共有=青、個人=グレー等）を徹底。エラー時は家族ユーザーに次の行動を優しく促すダイアログへフォールバックする。
- [x] **タスク管理・アクセシビリティ（a11y）Lintエラーの全解消**
  - **対策:** `biome check` で指摘されるアイコンボタンへの `aria-label` 不足などを解消し、スクリーンリーダー対応を完了させる。
- [ ] **モバイル向けタグ入力UIの「1タップサジェスト」導入**
  - **課題:** `tag-input.tsx` のCombobox型サジェストは、モバイルでソフトウェアキーボードが立ち上がると画面の半分を占有し、サジェスト一覧が隠れて操作性が著しく低下する。
  - **対策:** 入力欄の直下に、過去の利用頻度が高いタグ（`availableTags`）を「横スクロール可能なチップス」としてインライン表示し、キーボードを開かずに1タップで追加・削除できる親切なUIを併設する。
- [ ] **ロック解除モーダル閉鎖後の「フォーカス復元」制御**
  - **課題:** マスターキー未保持状態で「ヒント表示」などのアクション時にパスコード解除モーダルが立ち上がるが、解除後にフォーカスがページの先頭にリセットされ、ユーザーが操作位置を見失う。
  - **対策:** モーダル起動直前にフォーカスされていた要素（ボタン等）の `HTMLElement` を記憶し、モーダル閉鎖時に自動で `.focus()` を戻すフォーカス管理を実装。高級ホテルのような迷わせない丁寧な動線を作る。

## 💡 Phase 3: 日常利便性の向上とコミュニケーション支援 (Product UX)

「毎日ストレスなく情報を引き出せるか」にフォーカスし、情報の陳腐化や検索ノイズを防ぎ、エッジケースでもユーザーのデータを確実に救済する機能を実装します。

- [x] **家族招待リンクの「有効期限付コード」および「承認制」フローの導入**
  - **機能:** 家族の招待コード（`inviteCode`）に有効期限（例: 24時間）を設定。または、コード入力後に既存の家族管理者が「参加申請を承認」するまで加入を制限する。
  - **目的:** 招待URLやコードが意図せず流出した場合でも、無関係な第三者が自動的に家族のプライベートスペースへ潜り込むリスクをブラウザ・バックエンドレベルで完全に防ぐ。
- [ ] **無操作時の「E2EEオートロック機能」（Inactivity Timeout）**
  - **機能:** 一定時間（例: 5分）ユーザーの操作（マウス、タッチ、キー入力）がない場合、またはアプリが完全に裏側に回った（Visibility APIでの検知）際、自動でメモリ上の `masterKey` のステートを破棄（`null`化）してロック画面へリダイレクトする。
  - **目的:** 共有端末や外出先でのスマートフォン置き忘れ、離席時の覗き見による、意図しないデータの露出や盗み見を防ぐ。
- [ ] **致命的なデータ喪失を防ぐ「リカバリーキット（復元コード）」の発行**
  - **機能:** 家族パスコードや、パスコードを忘れた場合にマスターキーを再アンラップできるランダムな英数字の「リカバリー・キー（復元コード）」を生成し、QRコード付きのA4印刷用PDFとして出力・ダウンロードさせる機能。
  - **目的:** パスコード忘れによる完全なデータロスト（E2EEの宿命）を、アナログ（紙の金庫保管）での救済措置を設けることで回避し、非技術者ユーザーへ心理的安心感を与える。
- [x] **生体認証（WebAuthn / Passkeys）によるローカルロック解除**
  - **機能:** 初回入力時にキーをセキュア領域に保存し、次回以降は指紋/FaceIDでロック解除する。
  - **目的:** モバイルで毎回8文字以上のパスコードを入力する手間と、ショルダーハッキングのリスクを排除。
- [ ] **生体認証（WebAuthn / Passkeys）非対応デバイスへの配慮とフォールバック**
  - **機能:** `src/lib/biometric.ts` でPRF拡張非対応デバイス時に「高度な暗号化保護に対応していません」とハードエラーを返す挙動を改善。生体認証の登録フロー前にバックグラウンドで対応チェックを済ませ、非対応であれば親切なメッセージと共に最初から「パスコード手入力」へ綺麗にフォールバックさせる。
  - **目的:** 非対応端末を持つユーザーが、顔・指紋スキャンを通過した直後にエラーで突き放されるストレスフルなUXを排除する。
- [ ] **アクセス＆データ操作の「変更履歴（Audit Log）」と「最終更新者」の表示**
  - **機能:** 各レコードのメタデータとして「誰がいつヒントを閲覧したか」「誰が情報を更新したか」の履歴（監査ログ）をバックエンドで追跡・蓄積し、タイムラインとして表示する。また一定期間未更新のレコードには警告表示を出す。
  - **目的:** 家族間におけるデータ運用の透明性を高め、情報更新のトラブルや古いパスワードによるログイン失敗を未然に防ぐ。
- [ ] **ヒントの「更新リクエスト（SOS）」機能**
  - **機能:** 「ログインできない」ボタンを押し、対象レコードに「要更新」バッジを付与してオーナーへ通知する。
  - **目的:** 「パスワード変えた？」という家族間のやり取りをアプリ内で非同期に完結させる。
- [ ] **「ピン留め」によるクイックアクセス機能**
  - **機能:** 任意のレコードを最上部に固定する機能（`isPinned`）。
  - **目的:** 自宅Wi-Fiなど、頻繁に参照する情報の即時アクセス。
- [ ] **視覚的認知を高める「ファビコン表示」と「フォールバックアイコン設定」**
  - **機能:** フロントエンドで復号したURLからファビコンを取得して表示。内部ネットワークやOGP未設定サイト向けに、カスタム絵文字やカラーを設定できるUIを提供。
  - **目的:** テキストを読まずとも、ロゴの色や形で直感的にサービスを探せるようにする。
- [ ] **「パーソナル / 共有」のクイックフィルター**
  - **機能:** 画面上部で「すべて / 自分 / 共有」をワンタップで切り替えるトグル（デフォルトは「自分＋共有」）。
  - **目的:** 配偶者の個人的なサービス等、日常的に不要なノイズを隠す。
- [ ] **パスワードヒントの「テンプレート / アイデア」サジェスト**
  - **機能:** 「ペットの名前＋記念日」などのヒント例を入力フォームにサジェストする。
  - **目的:** 「何をヒントにすればいいか分からない」ユーザーへの登録支援。
- [ ] **タグの統合・一括リネーム機能**
  - **機能:** 「動画」と「Netflix」などの表記揺れタグを管理画面から一括マージ・置換する。
  - **目的:** 運用長期化に伴うタグの乱立・検索性の低下を解消。
- [ ] **データの破壊を防ぐ「アーカイブ（非表示）」機能**
  - **機能:** 削除の代わりに非表示（`isArchived`）にし、専用の一覧に退避する。
  - **目的:** ダッシュボードをクリーンに保ちつつ、「いつか使うかも」というユーザー心理に配慮する。
- [ ] **CSVインポート時の「プレビュー＆重複排除（Diff検証）」**
  - **機能:** CSVパース時に既存データと突合し、「新規/上書き/スキップ」のDry-run結果を提示する。また、大量データインポート時のスクロール負荷を下げるため、フロントエンドに `@tanstack/react-virtual` によるDOM仮想化スクロールを適用する。
  - **目的:** 誤操作によるデータの二重登録・破壊を防止し、数百件のデータ表示でもモバイルが一切カクつかない超高パフォーマンスな描画を実現する。
- [ ] **セッション切れ時の入力データ救済機能**
  - **機能:** 長文のメモやヒントをフォームに入力している最中にセッション（14日間）が切れ、保存ボタンを押した際に401エラーで入力内容が消滅してしまうリスクを完全に排除する。
  - **対策:** API通信（Mutation）の認証エラーをインターセプトし、入力ステートを保持したまま再ログインモーダルをポップアップ。再ログイン成功後に自動で保存処理を再開・完結させるグレースフル・デグラデーションを実装。
- [ ] **同時編集を考慮したUI/UX**
  - **機能:** Convexのリアルタイム性を活用し、編集中ユーザーの情報をDBで管理し、家族メンバーの同時編集時のuiに活用する

## 🚀 Phase 4: エコシステム拡張と将来への布石 (Ecosystem & Future-proofing)

デバイスや状況を問わないアクセス性と、マルチ展開を見据えて堅牢なテスト・ローカルキャッシュ機構を確立します。

- [ ] **PWA化による「完全オフライン閲覧対応」**
  - **機能:** Service Workerと IndexedDB を本格導入し、アプリの静的アセットだけでなく、Convexから同期済みの「暗号化された状態のレコードデータ」をローカルにセキュアにキャッシュする。
  - **目的:** 電波の届かない地下スーパーのレジ前、飛行機内、引越し直後などの完全なオフライン環境であっても、デバイス生体認証（またはパスコード）をパスすればローカルで安全に復号してヒントを確認できる、極上の可用性と実用性を提供する。
- [ ] **Convexの `getRecords` クエリにおけるDBインデックス最適化（B面アプローチ）**
  - **課題:** 現在の `getRecords` は `collect()` で全件取得した後にNode.jsメモリ上で配列のフィルター・ソートをかけている（データ肥大化時にメモリとCPUを過剰消費する $O(N)$ のインメモリボトルネック）。
  - **対策:** 検索用のタグ・タイトルトークン配列やインデックスフィールドをスキーマへ定義し、Convexネイティブの `.withIndex().paginate()` を活用した高速なページネーション＆絞り込み（$O(1)$ 近傍）へ移行する。
- [ ] **開発者向け暗号設計ドキュメント・シーケンス図の整備**
  - **機能:** `code-design.md` に、「マスターキー生成 → パスコードによるラップ/アンラップ → DB保存 → 生体認証（PRF）拡張による鍵導出」および「エンベロープ暗号（KEK/DEK構造）」の一連の流れを記述したMermaidシーケンス図や解説資料を新設する。
  - **目的:** 開発チーム全体の暗号文脈の理解度を統一し、将来的な機能拡張やメンテナンス時のセキュリティデグレード（バグによる脆弱性の混入）を未然に防ぐ。
- [ ] **複数家族（マルチ・ワークスペース）対応**
  - **機能:** 1ユーザーが複数の家族グループに所属し、ヘッダー等で切り替え可能にする。
  - **目的:** 「自分の現在の家族」と「実家の両親の家族」など、多様なコミュニティ構造へ適応させる。
- [ ] **コアロジック（Crypto/鍵管理）のnpmパッケージ化**
  - **機能:** `src/lib/crypto.ts` を `@poohma/crypto-core` のような独立パッケージに切り出す。
  - **目的:** 将来的なCLIツールやモバイルアプリ（React Native等）展開時のロジック再利用性の向上。
- [ ] **DB RLSとAPI連携のインテグレーションテスト拡充**
  - **機能:** pgTAPやVitestを用いたクロスファミリーのデータ参照（権限昇格）ブロックのCI検証。*(※一部対応済)*
- [ ] **重要暗号化フローのPlaywrightによるブラウザE2Eテスト導入**
  - **機能:** 実際のブラウザ環境における `crypto.subtle` やWebAuthn（PRF拡張モック）の挙動、E2EE暗号化・復号の一連のコアフローが、今後のアップデートによって破壊されないかを自動検証する。
  - **目的:** Node.js環境（JSDOM）の単体テストだけでは担保しきれない、ブラウザの実挙動に即した絶対的なセキュリティ品質を保証する。
